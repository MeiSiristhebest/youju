# 有据 - 领域术语表 / CONTEXT.md

## 项目背景

**有据** 是一个 AI 工作台，帮助用户在关键动作前（报名、签约、提交、购买）将分散的信息整合，找出冲突和缺失。

## 核心术语

### 材料 (Source)
用户上传的原始依据，包括：
- **聊天记录** (Chat)：微信/IM 中的沟通记录截图或文字
- **文档** (Document)：PDF、Word 等正式文件
- **网页** (Web)：从 URL 抓取的网页内容
- **截图** (Screenshot)：手机截图，OCR 识别提取文字
- **合同** (Contract)：正式签署的合同文本

### 冲突 (Conflict)
指不同材料之间的信息不一致，分为两种：

1. **直接矛盾** (Direct Conflict)：不同来源说了相反的事
   - 例：HR口头说"包水网"，合同写"水费自理"
   - 例：offer说"月薪25k"，合同写"年薪30万"

2. **承诺未落文字** (Undocumented Promise)：口头有承诺但正式文件没写
   - 例：HR口头说"每年调薪两次"，但offer没写
   - 例：中介承诺"家电坏了随时修"，合同没提

### 对齐 (Alignment)
将多个材料中的同一件事（截止时间、金额、条款等）提取出来，放在一起比较，标记出一致/冲突/缺失的状态。

### 风险 (Risk)
系统分析后输出的问题提示，分三个等级：
- 🔴 **严重风险** (Critical)：直接矛盾，可能导致重大损失
- 🟡 **待确认** (Pending)：承诺未落文字，需要向对方书面确认
- 🟢 **已对齐** (Aligned)：多源信息一致，无问题

### 话术 (Draft)
根据检测到的冲突，AI 自动生成的向对方确认/澄清的文字。用户可以复制后通过微信/邮件发送，留作书面凭证。

## 用户旅程

1. **上传材料** → 用户把聊天截图、文档、链接等丢进工作台
2. **AI 分析** → 系统提取关键要素（日期、金额、条款、责任方）
3. **冲突检测** → 跨源比对，找出矛盾和缺失
4. **报告输出** → 生成红/黄/绿灯风险看板 + 检查清单
5. **话术生成** → 针对每个风险点生成确认话术
6. **执行动作** → 用户带着话术去和对方确认，然后执行实际操作

## 设计原则

1. **不替用户做决定**：只展示风险，决策权归用户
2. **对齐才算完成**：必须所有关键项都绿了才算分析完成
3. **话术留凭证**：话术用于向对方求书面确认，留证据

## 技术方向

- **Demo 形式**：完整前后端（React + Node.js）
- **AI**：先 Mock，后接入 Claude/GPT
- **账号**：第三方登录（微信/oAuth）
- **离线**：不需要
- **任务管理**：先单一任务，后续扩展多任务
