# 有据 PRD

> 基于 grilling session 2026-06-29 生成的 PRD

## Problem Statement

用户在进行关键动作（报名、签约、提交、购买）前，面临"信息太多但还是容易出错"的问题。具体表现为：
- 重要信息分散在聊天记录、截图、网页、文档、合同等多处
- 信息之间可能存在矛盾（口头承诺 vs 正式文本）
- 用户容易看漏、记错、用错版本
- 结果导致返工、踩坑、甚至金钱损失

用户需要一个工具，能在真正提交/签字/购买前，把所有依据"放回同一张桌上"，帮人做一次认真的排雷检查。

## Solution

**有据** 是一个 AI 工作台：
1. 用户上传多种类型的材料（聊天截图、文档、网页链接等）
2. AI 自动提取关键要素（截止日期、金额、条款、责任方）
3. 跨源比对，检测直接矛盾和承诺未落文字
4. 输出红/黄/绿灯风险看板 + 可执行的检查清单
5. 针对每个风险点生成确认话术，用户可复制后向对方书面确认

**核心原则**：不替用户做决定，只帮用户在决定前先看清风险。

## User Stories

### 核心流程
1. 作为用户，我希望上传多种格式的材料（截图、PDF、URL），以便把散落各处的依据集中管理
2. 作为用户，我希望 AI 自动提取关键信息（日期、金额、条款），避免自己手动整理
3. 作为用户，我希望系统检测"直接矛盾"（不同来源说了相反的事），以便发现潜在风险
4. 作为用户，我希望系统检测"承诺未落文字"（口头有承诺但正式文件没写），以便及时要求书面确认
5. 作为用户，我希望看到红/黄/绿灯的风险评级，以便快速判断当前状态
6. 作为用户，我希望系统生成确认话术，我可以复制后发给对方，留作书面凭证
7. 作为用户，我希望检查清单能勾选，以便知道还有哪些待确认项

### 材料输入
8. 作为用户，我希望粘贴聊天记录截图，系统能 OCR 识别文字
9. 作为用户，我希望上传 PDF/Word 文档，系统能解析文本内容
10. 作为用户，我希望输入网页 URL，系统能抓取页面内容
11. 作为用户，我希望拖拽上传文件，操作直观简单

### 冲突检测
12. 作为用户，当我上传的材料中同一日期有不同的值时，系统应标记为冲突
13. 作为用户，当我上传的材料中金额数字不一致时，系统应标记为冲突
14. 作为用户，当口头承诺（聊天记录）的内容没有出现在正式文件（合同/Offer）中，系统应标记为"承诺未落文字"
15. 作为用户，当我上传的材料缺少关键项（如截止日期、金额）时，系统应标记为缺失

### 结果呈现
16. 作为用户，我希望看到清晰的摘要（如"3处严重风险，2处待确认"）
17. 作为用户，我希望每个风险点都能看到来源材料
18. 作为用户，我希望系统告诉我风险项之间的关联（如"这条风险关联到材料A和材料C"）

### 话术生成
19. 作为用户，我希望针对每个冲突生成对应的澄清话术
20. 作为用户，我希望话术能直接复制使用，有礼貌且有效
21. 作为用户，我希望话术中包含具体的引用（引用对方说过的原话）

### 账号与协作
22. 作为用户，我希望用微信登录，操作简便
23. 作为用户，我希望我的材料和分析结果保存在云端，换设备也能访问
24. 作为用户，我希望能把分析报告分享给相关方

### 任务管理
25. 作为用户，我希望一次只处理一个任务的核对（单一任务模式）
26. 作为用户，我希望能看到我历史提交过的任务记录

## Implementation Decisions

### 技术架构
- **前端**：React + TailwindCSS + Vite
- **后端**：Node.js + Express (或 Next.js API Routes)
- **数据库**：PostgreSQL（用户数据）+ Redis（Session）
- **文件存储**：S3/OSS（上传的文档）
- **AI 服务**：Claude API / GPT-4 API（上线后接入）

### 核心模块

#### 1. 材料解析器 (Source Parser)
- 聊天截图 → OCR 识别 → 文本提取
- PDF/Word → 文本解析
- 网页 URL → 内容抓取 + HTML 解析
- 输入框 → 直接文本

#### 2. 要素提取器 (Entity Extractor)
从文本中提取关键要素：
- **日期**：截止日期、有效期
- **金额**：薪资金额、费用、押金
- **条款**：责任方、赔付规则、交付物
- **承诺**：口头提及但未落文字的内容

#### 3. 冲突检测引擎 (Conflict Detector)
两种检测模式：
- **直接矛盾**：同类型要素（如金额）不同来源值不同
- **承诺未落文字**：A来源提到的内容在B来源（正式文件）中不存在

#### 4. 话术生成器 (Draft Generator)
基于模板 + AI 生成的混合模式：
- 针对每种冲突类型有预设话术模板
- AI 根据具体冲突内容填充变量
- 保持专业、有礼貌、便于对方回复的风格

#### 5. 报告生成器 (Report Generator)
输出内容：
- 风险总览（红/黄/绿灯数量）
- 详细风险列表（每条含来源材料和推荐动作）
- 可执行检查清单（用户可勾选）
- 对齐后的"统一版本参照"

### API 契约

```
POST /api/sources/upload
  → { sourceId, type, content }

POST /api/analyze
  → { risks: [{ type, level, sources, description, draft }] }

GET /api/report/:taskId
  → { summary, risks, checklist, alignedVersion }
```

### 数据模型

#### User
- id, email, thirdPartyId, thirdPartyType, createdAt

#### Task
- id, userId, title, status, createdAt

#### Source
- id, taskId, type, content, extractedEntities

#### Risk
- id, taskId, type, level, sourceIds, description, draftId

#### ChecklistItem
- id, riskId, text, checked

## Testing Decisions

### 测试策略
- **外部行为测试**：测试冲突检测结果，不测试内部实现
- **场景覆盖**：用真实用户场景（offer确认、租房签约）作为测试用例
- **边界测试**：空输入、单来源、多来源冲突等

### 测试用例示例
1. 上传2个材料，金额不同 → 应检测到直接矛盾（Critical）
2. 上传聊天记录提到"包水"，但合同没写 → 应检测到承诺未落文字（Pending）
3. 所有材料日期/金额/条款完全一致 → 应全部绿灯
4. 只上传1个材料 → 应提示信息不足，无法完成分析

## Out of Scope

以下功能在当前阶段不做：
- 离线模式（不需要）
- 多任务并行管理（先做单一任务）
- 实时协作编辑
- 高级数据分析报表
- 移动端原生应用
- 自动向对方发送邮件/消息

## Further Notes

### 设计哲学
- "有据不替你做决定，只帮你在决定前先看清"
- 分析结果不是结论，而是对话的起点
- 话术的价值在于"让对方书面确认"，留下凭证

### MVP 定义
当前已实现版本支持：
1. ✅ 手动输入/粘贴文本作为材料（聊天记录、文档、网页等）
2. ✅ 上传文件（TXT/PDF）作为材料
3. ✅ 输入网页 URL 自动抓取内容
4. ✅ AI 分析接口（配置 API Key 后使用真实 AI，未配置使用规则模拟）
5. ✅ 风险点检测（直接矛盾、承诺未落文字、信息缺失）
6. ✅ 检查清单（可勾选）
7. ✅ AI 生成确认话术
8. ✅ 单一任务模式（内存存储）

### 后续扩展方向
- 多任务管理 + 数据库持久化
- 截图 OCR 识别
- 第三方登录（微信/oAuth）
- 报告导出（PDF）
- 多用户协作
